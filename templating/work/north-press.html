Case study · Editorial platform

North Press

A 110-year-old editorial cooperative replaced their single-page web application with a 300-line build script.

Client
North Press Cooperative
Year
2026
Role
Design & engineering
Stack
Static HTML · Python · vanilla JS

Brief

North Press publishes a weekly long-form journal that has been in continuous print since 1916. Their previous site was a single-page React application that took twelve seconds to load on a slow connection and routinely confused readers in feed readers and search engines.

The brief was simple: keep the editorial workflow intact, but make every page legible to a browser, a feed reader, an LLM, and a print archivist — with one source of truth.

Approach

We replaced the SPA with a build script of about three hundred lines. Each article is a markdown file in the editorial repository, with a front-matter block describing the publication context.

The script reads the file, parses the prose, runs each figure through its renderer — charts compile to inline SVG, videos copy through — and emits a single static HTML page with a JSON-LD block in the head.

Build

The build step has no plugin system, no lifecycle, no hydration story. It is a single function that returns a string. Replacing it would be a half-day's work. The team can read it in an afternoon.

Outcome

On a slow 3G profile, time-to-first-byte fell from 1100ms to 90ms. More usefully, the custom React admin that the team had been maintaining alongside the SPA was retired. Editors now write in a folder of markdown files, on the same workflow they use for the print edition.

Reflection

What surprised us most was how cleanly the new pipeline mapped onto the team's existing print discipline. The website stopped being a separate kind of object — it became another output of the editorial process, like the print edition or the audio podcast.
